## Decisions Requiring Operator Input

### CLI backend product promise

This decision is really about product positioning around Anthropic's `claude-cli` path. The question is not whether Superhuman has to give up generic local CLI backends across the whole product. The question is whether Claude subscription-backed reuse should still be presented as a normal default onboarding path, or whether that needs to be narrowed now that upstream is backing away from it.

If we keep generic CLI support, the architecture can stay broadly the same, but onboarding, docs, and provider language need to stop implying that Anthropic subscription reuse is the default safe path.

- Option A: keep generic CLI support while narrowing Anthropic-specific claims.
  - Pros: preserves a real product differentiator and keeps the broader local-runtime story intact.
  - Cons: Anthropic-specific paths remain policy-sensitive and may continue to create support burden.
- Option B: follow upstream more closely and reduce CLI support overall.
  - Pros: reduces vendor-policy exposure and simplifies onboarding and provider positioning.
  - Cons: weakens the "use the local tool you already pay for" product promise and pushes more users toward API-only billing.
- Recommendation: keep generic CLI support, but stop positioning Claude subscription reuse as the default Anthropic path.
- Blocked until decided: final product language and implementation posture around vendor-specific CLI paths.

### Dreaming trust model

The open question here is how much authority dreaming should have. Upstream is exploring separate dream artifacts and staged promotion, but for Superhuman the real choice is whether dreaming stays a reviewable, plugin-owned memory assistant or becomes something closer to an automatic memory writer. This does not require shipping the full upstream dreaming product surface right now, but it does decide where the logic lives and how trustworthy the resulting memory behavior feels.

- Option A: keep dreaming plugin-owned, separate-artifact, append-only, and promoted-insight-first.
  - Pros: keeps provenance clearer, preserves reviewability, and avoids over-trusting synthesized memory too early.
  - Cons: delays the more magical feeling of stronger automatic memory behavior.
- Option B: move toward a stronger automatic memory writer with more direct promotion into first-class memory.
  - Pros: creates a more powerful and more seamless memory experience if it works well.
  - Cons: makes trust, auditability, and ownership boundaries much easier to blur.
- Recommendation: keep it plugin-owned, separate-artifact, append-only, and promoted-insight-first.
- Blocked until decided: final memory and dreaming implementation shape.

### Shared-core security follow-up

The main autonomous security call is straightforward: shared-core hardening should generally follow upstream here. The concrete gap is the `before_tool_call` failure path, which still looks fail-open locally and should be brought into the fail-closed shape. Stronger local hardening elsewhere does not need to be removed.

- Why this was safe to decide: current policy already favors upstream-shaped shared-core security fixes unless there is a real product-policy conflict.
- Next: patch hook failure behavior and reconcile duplicate local hardening carefully.

### Reply-dispatch genericization

The architectural direction is good, but this should be an adaptation rather than a blind copy. Superhuman should keep the current public `reply-dispatch-runtime` seam while aligning internals with upstream's generic plugin-owned interception model. The important non-change is that the public seam does not need to be renamed or deleted.

- Why this was safe to decide: it preserves compatibility and does not change product direction.
- Next: preserve the exported dispatcher contract while reducing hidden special casing internally.
